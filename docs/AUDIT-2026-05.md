# SysTUI — Security & quality audit (2026-05-26)

Independent audit of the v1.0.0 codebase on branch `security/audit`, assessing whether
the code is secure and optimal for use. It covers dependency advisories, the command-
execution/SSH paths, privilege and credential handling, file/path handling, and
robustness/performance. Two issues were found and fixed in this branch; the rest is a
clean bill of health. Complements the release [`SECURITY-REVIEW.md`](SECURITY-REVIEW.md).

## Method

- **Dependency advisories:** `cargo audit` against the RustSec database (239 crates).
- **Code review:** the execution choke points (`CommandSpec`, `LocalTransport`,
  `SshTransport`), the action engine, the database/credential layer, the audit log,
  config/state persistence, and the parsers.
- **Robustness:** static scan for production `unwrap`/`expect`/`panic`/`unreachable`,
  cross-checked against the S10.4 parser fuzzing.
- **Tooling:** `cargo fmt`/`clippy -D warnings`/`test` (361 tests) green after fixes.

## Findings

### F1 — SSH argument/option injection via destination *(Medium) — FIXED*

`SshTransport` passes the `user@host` destination as a positional argument to the system
`ssh` client. OpenSSH has **no `--` end-of-options marker**, so a host or user beginning
with `-` (e.g. `-oProxyCommand=touch /tmp/pwned`) would be parsed as an `ssh` option and
could execute an arbitrary **local** command — bypassing the project's "no free-form
command" guarantee at the argv layer. Destinations come from operator config/CLI, but a
shared or templated fleet inventory makes this a real defense-in-depth gap.

**Fix:** `SshTransport::check_destination` rejects a host/user with a leading `-` before
any process is spawned (`CoreError::InvalidInput`), enforced at the single `exec` choke
point that all remote run/read/list operations pass through. Covered by a new test.

### F2 — Audit log world-readable *(Low) — FIXED*

The append-only audit log (`~/.local/share/systui/audit.log`) records who did what on
which host. It was created with the default umask (often `0644`), so on a shared machine
other users could read the operator's action history.

**Fix:** the log is now opened with mode `0600` on Unix. (It contains action metadata
only — no secrets or command output — so this is hardening, not a leak of credentials.)

### F3 — Config/state files use default permissions *(Low) — documented, not changed*

`config.toml` (SSH inventory: hostnames, users, ports — **no passwords**) and the local
state store (`store.json`: health trends, session notes, saved searches) are written with
the default umask. They contain operational metadata, not secrets, and normally live in
permission-restricted user directories, so this is low risk. **Recommendation:** apply
the same `0600` treatment as the audit log if SysTUI is run on shared hosts. Left
unchanged to keep this audit's footprint minimal; trivial to add later.

### F4 — Transitive advisories in `ratatui 0.29` *(Informational)*

`cargo audit` reports **no vulnerabilities**. Two warnings, both transitive via
`ratatui 0.29` and not reachable as exploits from SysTUI:
- `paste 1.0.15` — unmaintained (RUSTSEC-2024-0436).
- `lru 0.12.5` — unsound `IterMut` (RUSTSEC-2026-0002); SysTUI never touches `lru`.

**Recommendation:** clear automatically on a future `ratatui` bump. Consider adding a
`cargo audit` (or `cargo deny`) step to CI to track new advisories.

## What was verified clean

- **No free-form commands / no shell.** `LocalTransport` runs `Command::new(program)`
  with an argv vector — never a shell. `SshTransport` POSIX-quotes program and every
  argument in one place (`quote_arg`), with a safe-character allowlist that excludes all
  shell metacharacters; the `/etc/passwd; rm -rf /` injection case is tested. The only
  `sh -c` (cron "run now") reproduces cron's own execution model with the crontab line
  as a single argv element — not free-form input.
- **No `unsafe`.** The workspace sets `unsafe_code = "forbid"`; no `unsafe` blocks exist.
- **No risky panics.** Exactly **two** `expect` calls in production code, both on
  infallible invariants (an unclosable semaphore; a constant regex). Parsers are fuzz-
  proven (S10.4) to never panic on adversarial input.
- **Privilege.** SysTUI never auto-escalates; it runs as the invoking user, and the
  action engine gates every mutation (mode + guardrail + typed confirmation + audit),
  enforced on both `plan` and `execute`.
- **Credentials.** The database layer is credential-free: it detects credential
  *sources* (socket, `.pgpass`, env) as labels and never stores or renders secrets.
- **Reports.** HTML reports escape host-derived text (`<`, `&`, etc.), preventing HTML/JS
  injection into generated reports.
- **Bounded growth.** The state store caps retention (90 snapshots / 50 notes / 30
  searches per host); logs are bounded by `journalctl -n`. No unbounded allocation.
- **Performance.** The v0.8.3 phase moved refresh off the UI thread, made collectors
  concurrent with per-collector timeouts (a slow host degrades to partial data, never a
  stall), and tiered slow-changing data; log parsing is linear (benchmarked, ~0.9M
  entries/s). Release profile uses fat LTO + strip.

## Verdict

The codebase is **secure and fit for use**. No exploitable vulnerabilities were found.
One Medium defense-in-depth gap (SSH option injection) and one Low hardening item (audit
log permissions) were fixed in this branch; remaining items are Low/Informational with
clear, optional follow-ups. The architecture's core safety guarantees (no free-form
commands, no `unsafe`, engine-gated privilege, credential-free discovery, graceful
degradation) hold under review.
